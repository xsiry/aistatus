# aistatus 使用说明手册

## 1. 项目简介

`aistatus` 是一个面向终端的多账号 Codex/OpenAI 配额状态面板，目标是以接近 `btop` 的信息密度展示：

- 多账号列表
- 5 小时额度使用百分比
- 周额度使用百分比
- 会员等级 / 账号类型
- 刷新状态 / 认证异常 / 协议兼容状态

当前实现优先支持 **Linux / macOS**，核心数据源为 **Codex 协议 v2**。

---

## 2. 核心概念

### 2.1 订阅额度 vs API Usage

本项目明确区分两类数据：

#### A. ChatGPT / Codex 订阅额度

这是主展示对象，包括：

- `5h` 百分比
- `weekly` 百分比
- `planType` / membership tier

#### B. OpenAI API Usage

这是另一条独立 usage family。

当前项目只预留了 `OpenAiApiUsageProvider` skeleton，**不会把 API usage 伪装成订阅额度**。

## 5. Profile 管理

### 5.1 查看账号列表

```bash
cargo run -- profile list --config .sisyphus/fixtures/sample-config.toml
```

### 5.2 设置默认账号

```bash
cargo run -- profile set-default acct-api-key --config .sisyphus/fixtures/sample-config.toml
```

### 5.3 添加账号

```bash
cargo run -- profile add \
  --config /tmp/aistatus-config.toml \
  --id acct-plus \
  --name "Primary ChatGPT" \
  --auth-mode browser \
  --account-kind chatgpt \
  --provider codex_protocol \
  --membership-tier plus \
  --plan-type plus
```

### 5.4 登录材料写入

#### keyring backend

```bash
cargo run -- profile login \
  --config /tmp/aistatus-config.toml \
  --id acct-plus \
  --auth-mode browser \
  --secret "session-cookie-value"
```

#### 加密文件 backend

```bash
cargo run -- profile login \
  --config /tmp/aistatus-config.toml \
  --id acct-api \
  --auth-mode api_key \
  --secret "sk-live-123" \
  --file-store
```

### 5.5 清除登录材料

```bash
cargo run -- profile logout acct-plus --config /tmp/aistatus-config.toml
```

### 5.6 非法 auth-mode 行为

如果传入未知 auth-mode，例如：

```bash
cargo run -- profile add --config /tmp/x.toml --id acct-bad --name Bad --auth-mode nonsense --account-kind chatgpt --provider codex_protocol
```

现在会明确失败：

```text
invalid auth mode `nonsense`
```

不会再静默回退成 `browser`。

## 6. Refresh 刷新命令

### 6.1 正常刷新

```bash
cargo run -- refresh --config .sisyphus/fixtures/sample-config.toml --force --now-epoch-secs 1000
```

输出会展示每个 profile 的刷新结果，例如：

- `healthy`
- `auth_required`
- `skipped`

### 6.2 手动刷新策略

如果某个 profile 配置了：

```toml
allow_manual_refresh = false
```

则强制刷新会被共享刷新路径拦截，并输出：

```text
manual refresh disabled by policy
```

这条规则同时约束：

- CLI `refresh --force`
- TUI 中按 `r`

### 6.3 认证材料缺失行为

若 profile 使用 `Managed` secret ref，但当前没有 sidecar material 或真实 backend 可读内容，刷新不会再误报 `healthy`，而会输出 `auth_required` 或相应失败原因。

## 7. TUI 使用

### 7.1 启动 fixture TUI

```bash
cargo run -- tui --fixtures sample-quotas
```

这会进入真实交互式终端界面。

### 7.2 支持的快捷键

- `j` / `↓`：下一个账号
- `k` / `↑`：上一个账号
- `a`：切换到账户详情视图
- `d`：切换到诊断视图
- `r`：触发共享 refresh 路径并显示 refresh 反馈
- `?`：显示 / 隐藏帮助
- `q`：退出

### 7.3 当前 TUI 显示内容

账号列表区会显示：

- display name
- provider
- status / health
- last refresh age
- 5h 摘要
- weekly 摘要
- membership

详情区会显示：

- account kind
- membership
- provider
- auth mode
- refresh interval
- manual refresh 是否允许
- protocol compatibility
- last error
- account / diagnostics 两类视图内容

## 8. 配置与 Secret 说明

### 8.1 Plain config 保存什么

纯配置文件保存：

- `default_profile_id`
- profile 元数据
- `SecretRef`

不保存：

- API key 明文
- browser/headless session payload

### 8.2 Secret 保存位置

secret payload 可以来自：

- native keyring
- 加密文件 fallback
- sidecar `SecretMaterial` JSON（开发 / CI 用）

### 8.3 安全性补充

项目已经额外做了以下健壮性处理：

- 损坏加密文件的 nonce 长度会被拒绝，不会 panic
- 非法 auth-mode 会显式报错，不会被默默接受
- 缺 secret/backend 时 refresh 不会再盲成功

## 10. 协议兼容策略

- 当前 Codex 适配层钉死到 `schema v2`
- schema 漂移时直接报 incompatibility
- 已知 `planType` 会归一化为稳定 membership buckets
- 未知 `planType` 会保留 `raw_plan_type`

---

## 11. 不支持项

当前明确不支持：

- Windows
- 抓取 ChatGPT 网页私有接口
- 把 OpenAI API usage 伪装成订阅额度
- 自动联动 OpenCode/Codex 外部会话切换
- 后台 daemon / 远程同步

## 14. 常见问题

### Q1：为什么我会看到 `auth_required`？

通常表示：

- session 失效
- sidecar secret 缺失
- keyring / file backend 读不到 secret

### Q2：为什么 `refresh --force` 没执行而是显示 skipped？

可能该 profile 配置了：

```toml
allow_manual_refresh = false
```

### Q3：为什么传错 `--auth-mode` 直接报错了？

这是有意加固后的行为，防止错误输入被默默解释成 `browser`，造成配置污染。

### Q4：为什么 TUI 里的 `r` 不一定把所有账号都刷新成成功？

因为 TUI 复用了共享 refresh 路径，真实行为会受：

- auth 状态
- secret/backend 可用性
- manual refresh policy

共同影响。
